## 2026-05-03

### 可用性评估

当前项目已经具备可运行 MVP 能力，真实 Playwright 执行链路已验证通过：

```text
execution passed=true mode=playwright
```

系统适合为中小型 Web 项目自动生成第一批 L0 / smoke E2E，发现页面白屏、路由错误、接口 5xx、基础 CRUD 断裂、测试模板不稳定等问题。

但系统还不能承诺“随便一个项目都能测出 bug”。真实业务 bug 依赖业务 oracle，也就是系统必须知道什么结果算正确。后续演进需要让目标项目显式提供测试环境、登录方式、种子数据、接口契约和业务断言规则。

### 已修复问题

- 修复 `test_artifacts.id` 全局主键冲突。生成器会为不同任务生成相同相对路径的 artifact，例如 `e2e/playwright.config.ts`；保存层现在使用 `taskID + artifactID` 作为数据库行 ID，允许不同任务保存同名 artifact。
- 新增回归测试覆盖不同任务保存相同 artifact ID，以及同一任务重复保存时只替换当前任务的 artifact。

### 下一阶段路线

下一阶段按真实项目接入能力推进，而不是继续堆 demo 专用逻辑：

1. 目标项目配置：读取 `.autotest.yaml` / `.autotest.yml`，支持 `base_url`、启动/seed/reset 命令、认证信息、安全黑名单和业务断言声明。
2. OpenAPI / Swagger 优先：有接口契约时，优先用契约生成请求 payload、状态码断言和响应 schema 校验。
3. 认证与数据初始化：支持测试账号登录、storage state、seed/reset 钩子。
4. 强断言升级：把弱断言升级为创建后可查询、字段一致、非法输入返回 400、未授权返回 401/403 等业务断言。
5. Bug-seeking 模式：主动探测空值、超长、非法枚举、越权访问、重复提交、删除后读取、分页边界等问题。
6. 失败诊断与修复：继续优化多文件失败诊断、模板级补丁生成和 Review Guard。

## 2026-04-28

### 通用化修正

本轮修正明确了系统与 `test-app` 的边界：

- `test-app/` 仅作为样例目标应用，不再作为生成逻辑的隐含模板。
- Playwright 生成器去除了 `/api/notes`、`Title`、`Content`、`Status` 等样例应用专用假设。
- 生成器改为基于 `api.scan` 和 `db.scan` 的结果推导测试：扫描到 `POST` 接口和数据模型时，会生成应用级数据写入验证。
- payload 按字段名自动生成，跳过常见服务端字段，如 `id`、`created_at`、`updated_at`。
- 写入后优先通过同集合 `GET` 接口回查，验证业务数据对目标应用真实生效。

### 数据库语义澄清

系统现在明确区分两类数据库：

- `.autotest/autotest.db`：自动化系统元数据库，只保存任务、事件、Agent Run、执行记录和报告索引。
- 目标应用数据库：由被测应用自己使用。测试里的“入表”指通过目标应用 API/UI 触发业务写入，让数据进入目标应用的隔离测试数据库。

用户需要负责将目标应用测试库与真实生产库隔离；自动化系统会把该测试库当作真实业务环境验证，而不是把业务数据写入系统元数据库。

### 测试与修复状态

- 失败修复分支中，补丁应用后如果重跑通过，任务最终状态现在为 `SUCCEEDED`。
- 如果诊断完成但重跑仍失败，才保留为 `PARTIAL_SUCCEEDED`。
- 新增测试覆盖通用数据写入生成逻辑，确保不会继续生成 `test-app` 专用的 `/api/notes` 路径。
- 新增测试覆盖修复重跑通过后的最终状态。

### 本次修正

真实 Playwright rerun 的失败点来自生成模板中的错误断言：

```text
scenario-summary should contain String(scenario.steps.length)
```

但页面 fixture 并没有渲染这个数字，因此该断言天然不稳定。现在生成器已改为只校验 `scenario.steps.length > 0`，避免继续生成无法通过的 DOM 断言。

随后又修复了多条 precondition 的断言方式：不再把列表项用空格拼成一个整体字符串，而是逐条断言每个 precondition 是否出现在 DOM 中。
